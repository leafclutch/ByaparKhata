## Project Overview

**HamroHisab** is a multi-tenant SaaS business operations platform built for Nepal-based retail businesses. It provides inventory management, billing, sales/purchase tracking, expense recording, and analytics.

- **Company**: Leafclutch Technologies Pvt. Ltd.
- **Target users**: Small-to-medium retail businesses in Nepal
- **Deployment**: Vercel (production at ap-northeast-2 Supabase project)
- **Business model**: Subscription-based SaaS — Superadmin manages company subscriptions

### User Roles

| Role | Scope | Access |
|------|-------|--------|
| `superadmin` | Platform-wide | All companies, user management, subscription management |
| `admin` | Company-scoped | Analytics and reports only (read-only) |
| `operator` | Company-scoped | Full CRUD — sales, purchases, expenses, inventory, products |

Role is stored in Supabase JWT `user_metadata.role`. Company membership is in `user_metadata.company_id`.

## Business Rules

### Tenant Isolation
- Every data table has a `company_id` column.
- RLS policies use `get_user_company_id()` DB function which reads from `users.company_id` for the authenticated user.
- A company user can **never** access another company's data at the DB level.
- Superadmin bypasses company isolation using the `service_role` key via server-side API routes only.
- The middleware (`proxy.ts`) also checks `company_status` on every protected request and redirects to `/subscription-expired` if the company is paused/disabled/expired.

### Inventory Rules
- Products are **soft-deleted** (`is_active = false`), never hard-deleted.
- Every stock change (sale, purchase, manual adjustment) writes a row to `inventory_transactions` with `reference_type`, `reference_id`, `user_id`, and optional `notes`.
- `inventory_transactions` is the authoritative audit trail for all stock movements.
- Products have optional expiry tracking: `manufacture_date`, `expiration_date`, `expiry_notification_days`.
- A DB trigger (`notify_low_stock`) fires when product quantity drops at or below `min_stock` and inserts a notification row.
- A pg_cron function `generate_expiry_notifications()` runs daily to insert expiry alerts.
- **⚠ Known DB/App Conflict**: `schema.sql` has triggers (`sale_items_decrement_stock`, `purchases_increment_stock`) that auto-update stock on insert. The application services **also** update stock manually. If the triggers are active, stock may be updated twice. Verify trigger status in Supabase before enabling both. The recommended approach is to rely on app-level updates and disable the schema triggers.

### Sales Rules
- Invoice number format: `HH-YYMM-XXXX` (e.g. `HH-2606-1234`) — generated in `generateInvoiceNumber()`.
- A sale has a parent record (`sales`) and child rows (`sale_items`).
- Deleting a sale restores all product stock and logs a `return` inventory transaction.
- Updating a sale restores old stock, validates new stock availability, then re-deducts.
- `subtotal = sum(item.qty × item.unit_price - item.discount)`, `grand_total = subtotal - order_discount`.
- Sales can include remarks (`notes` field) and are searchable by remarks.

### Payment Rules
- **Supported methods**: `cash`, `online`, `mixed`
- `card` and `bank_transfer` are **not supported** and must not appear in any dropdown.
- **Mixed payment**: When `payment_method = "mixed"`, both `cash_amount` and `online_amount` must be stored. Their sum **must equal** `grand_total` — enforced in the UI before save.
- The DB `sales` table has `cash_amount` and `online_amount` nullable columns (added via `migration_payment_remarks.sql`).
- **⚠ DB constraint mismatch**: The original `schema.sql` has `CHECK (payment_method IN ('cash','upi','card','bank_transfer'))`. This constraint must be dropped/updated when running migrations on a fresh DB to support `'online'` and `'mixed'`.

### Purchase Rules
- Purchases are against a single product per record (no multi-line purchase orders).
- Purchases increase product stock; deleting a purchase decreases stock.
- Purchases support optional remarks (`notes`).

### Invoice / Billing Rules
- Invoice preview and printing is at `/operator/billing?sale=<id>`.
- Print uses `window.print()` with CSS class `.print-invoice` to isolate the invoice DOM.
- Elements with class `.no-print` are hidden during printing.
- The invoice shows company logo (or company name if no logo), address, PAN/VAT, payment breakdown, items table, totals, and remarks.
- Mixed payment shows `Cash: Rs. X` + `Online: Rs. Y` in the payment section.

### Analytics Requirements
- Admin dashboard shows MTD KPIs with month-over-month percentage change.
- Revenue is calculated as: `net_profit = total_sales - total_purchases - total_expenses`.
- `getDashboardKPIs` makes 7 parallel Supabase queries to compute all KPIs in one pass.
- `getMonthlyRevenue` runs 6 months in parallel (not sequential) using `Promise.all`.
- Operator performance stats are derived from `sales` + `purchases` + `activity_logs`.

### Routing Architecture

**Middleware zones** (defined in `src/proxy.ts`, invoked by `src/middleware.ts`):

```
Zone 1: /superadmin/*
  └─ Validates JWT role === "superadmin"
  └─ /superadmin/login: redirect → /superadmin/dashboard if already SA

Zone 2: /admin/* and /operator/*
  └─ Validates authenticated Supabase session
  └─ Role mismatch: admin → /operator redirect, operator → /admin redirect
  └─ Superadmin trying company routes → /superadmin/dashboard
  └─ Company status check (5-min httpOnly cookie cache "co_ok")
     └─ paused/disabled/expired → /subscription-expired

Public: /login, /auth/callback, /auth/reset-password, /subscription-expired
```

**Admin routes** (analytics only — no CRUD):
`/admin`, `/admin/revenue`, `/admin/analytics`, `/admin/analytics/categories`, `/admin/analytics/operators`, `/admin/profit-loss`, `/admin/reports`, `/admin/inventory-insights`, `/admin/notifications`, `/admin/settings`, `/admin/activity`

## Database Documentation

### Table Relationships

```
companies
  ├── users (company_id)
  ├── categories (company_id, parent_id self-ref)
  ├── products (company_id, category_id → categories)
  ├── sales (company_id, operator_id → users)
  │     └── sale_items (sale_id → sales, product_id → products)
  ├── purchases (company_id, operator_id → users, product_id → products)
  ├── expenses (company_id, operator_id → users)
  ├── notifications (company_id)
  ├── invoices (company_id, sale_id → sales)
  ├── subscription_renewals (company_id)
  └── activity_logs (company_id, user_id)

inventory_transactions (company_id, product_id → products, user_id)

superadmin (id → auth.users)
```

### Important Table Notes

**`sales`** (beyond schema.sql):
- `cash_amount` NUMERIC(12,2) — populated when payment_method = 'mixed'
- `online_amount` NUMERIC(12,2) — populated when payment_method = 'mixed'
- `notes` TEXT — operator remarks

**`inventory_transactions`** (beyond schema.sql):
- `reference_type`: `'purchase' | 'sale' | 'return' | 'adjustment'`
- `reference_id`: UUID of the triggering sale/purchase (null for manual adjustments)
- `user_id`: who made the change
- `notes` TEXT — reason for manual adjustments

**`activity_logs`** (added by migration_activity.sql):
- `user_id`, `user_name`, `user_role` — snapshot of actor at time of action
- `action`: `'create' | 'update' | 'delete' | 'adjustment'`
- `entity_type`: `'sale' | 'purchase' | 'product' | 'expense' | 'category'`
- `entity_id` TEXT, `entity_label` TEXT — human-readable reference

**`schema.sql` legacy defaults (not yet updated)**:
- `companies.currency DEFAULT 'INR'` — should be NPR (UI uses NPR, DB default is legacy)
- `companies.timezone DEFAULT 'Asia/Kolkata'` — should be Asia/Kathmandu
- `payment_method CHECK (... 'upi', 'card', 'bank_transfer' ...)` — must be updated for mixed payment support

### RLS Strategy

All data tables have RLS enabled. Two helper functions live in the DB:
- `get_user_company_id()` — `SELECT company_id FROM users WHERE id = auth.uid()`
- `get_user_role()` — `SELECT role FROM users WHERE id = auth.uid()`

Generic company isolation policy on all data tables:
```sql
CREATE POLICY "company_isolation" ON <table>
  USING (company_id = get_user_company_id());
```

Superadmin API routes use `service_role` key (bypasses RLS) via `createClient` from `src/lib/supabase/admin.ts`. Never expose service_role to the browser.

### Tenant Isolation Strategy

1. **DB level**: RLS via `get_user_company_id()` — all queries filtered to caller's company.
2. **App level**: Services always pass `company_id` from the authenticated user object.
3. **Middleware level**: `proxy.ts` enforces role-based routing — admin cannot reach operator routes and vice versa.
4. **Superadmin separation**: SA routes and data are completely isolated. SA users are stored in the `superadmin` table (not `users`). SA never accesses company routes.

### Inventory Transaction Strategy

Every stock change must write to `inventory_transactions`:
- **Sale created** → `reference_type = 'sale'`, `quantity_change = -qty`
- **Sale deleted** → `reference_type = 'return'`, `quantity_change = +qty`
- **Sale updated** → old items get `'return'`, new items get `'adjustment'`
- **Purchase created** → `reference_type = 'purchase'`, `quantity_change = +qty`
- **Purchase deleted** → `reference_type = 'adjustment'`, `quantity_change = -qty`
- **Manual stock adjust** → `reference_type = 'adjustment'`, `notes` contains reason

`inventory_transactions` is the single source of truth for all stock movements.

## Important Decisions

### Why `proxy.ts` instead of `middleware.ts`
Next.js 16 uses `proxy.ts` as the routing middleware file. `src/middleware.ts` exists as the entry point but delegates entirely to `proxy()` from `src/proxy.ts`. The proxy implements 3 routing zones with clean separation between SA and company user flows.

### Why `inventory_transactions` is the source of truth
The `products.quantity` column is the current stock count. `inventory_transactions` is the immutable ledger of every change that produced that count. This enables full stock movement audits, operator accountability, and the ability to reconstruct quantity from the transaction log. Every mutation (create/delete/update sale or purchase, manual adjustment) must write to this table.

### Why tenant isolation is both at RLS and app level
RLS alone is sufficient for security. The app-level `company_id` filtering is belt-and-suspenders — it also prevents accidental cross-tenant data in edge cases during development and makes query intent explicit in code reviews.

### Why payment methods are Cash/Online/Mixed only
Nepal retail businesses predominantly use cash and QR/mobile banking (collectively "Online"). Card terminals are rare. Bank transfer is used for wholesale but the platform abstracts this as "Online". "Mixed" covers the very common scenario where a customer pays part cash, part online. Simplicity over exhaustiveness.

### Why mixed payment validation is client-side
The DB stores `cash_amount` and `online_amount` as nullable columns without a CHECK constraint ensuring they sum to `grand_total`. The enforcement is in the UI (`mixedBalance === 0` check) before any save operation. This is pragmatic — a DB constraint would require a complex multi-column check.

### Why there is no global data store
All page data is loaded fresh on mount via `useEffect`. There is no React Query, SWR, or global data cache. This keeps the architecture simple for a CRUD-heavy app where stale data is worse than a loading spinner. The only global state is UI state (sidebar) via Zustand.

### Why invoice printing uses `window.print()` and not PDF generation
Browser-native print is zero-dependency, always produces accurate layout, and supports the operating system's PDF export. A server-side PDF generator would add complexity and cost without meaningful benefit for this use case.

### Why admin cannot access operator routes
Admin is strictly analytics/read. The middleware (`proxy.ts`) redirects admin users trying to reach `/operator/*` back to `/admin`. The AdminSidebar only links to `/admin/*` routes. This enforces the business rule that admin cannot create/edit/delete transactions.

### Why `formatNPR` uses `en-IN` locale for number grouping
Nepal uses the same lakh/crore grouping system as India (1,00,000 not 100,000). The `en-IN` locale produces correct comma placement. The `Rs.` prefix is added manually to avoid browser/environment inconsistency in rendering the NPR currency symbol.

### Why activity logs use user name snapshots
`activity_logs.user_name` stores the name at the time of the action. If a user is renamed later, historical logs remain accurate. This is a deliberate denormalization for audit integrity.

## Known Issues / Technical Debt

### Critical
1. **DB trigger vs app-level stock conflict**: `schema.sql` has triggers `sale_items_decrement_stock` and `purchases_increment_stock` that auto-update stock on insert. Services also update stock manually. If triggers are active on the DB, stock will be double-updated. **Resolution**: Drop the two stock triggers from the DB and rely exclusively on application-level stock management.

2. **Payment method CHECK constraint mismatch**: `schema.sql` has `payment_method CHECK (payment_method IN ('cash','upi','card','bank_transfer'))` on `sales`, `purchases`, and `expenses` tables. The frontend uses `'cash' | 'online' | 'mixed'`. New inserts with `'online'` or `'mixed'` will fail if the constraint is active. **Resolution**: Run an ALTER TABLE to update the constraint.

### Schema Defaults Not Updated
- `companies.currency DEFAULT 'INR'` — should be `'NPR'`
- `companies.timezone DEFAULT 'Asia/Kolkata'` — should be `'Asia/Kathmandu'`
- These affect new company creation defaults only; existing records are fine.

## Claude Working Rules

When working on this project:

1. **Always read `.claude.md` first.** It contains decisions, constraints, and context that are not obvious from code alone.

2. **Fix root causes, not symptoms.** If a bug exists because of an architectural issue, fix the architecture. Do not patch around it.

3. **Never introduce hardcoded data.** No mock objects, no demo constants, no placeholder arrays in production code. All data comes from Supabase.

4. **Never use IS_DEMO_MODE or similar fallback patterns.** This was a historical anti-pattern that has been removed. The app always uses real Supabase.

5. **Maintain strict tenant isolation.** Every DB insert must include `company_id`. Every query must filter by `company_id`. Never mix company data.

6. **Preserve Nepal localization.** All currency displays must use `formatNPR()`. No `₹` symbol. No INR. No India references. Timezone is `Asia/Kathmandu`. Payment methods are Cash, Online, Mixed only.

7. **Remove dead code when discovered.** No unused imports, no unused state, no unreachable branches.

8. **Keep architecture clean.** Services handle DB access only. Pages handle UI only. No business logic in components. No DB queries in UI components.

9. **Activity logging is fire-and-forget.** `logActivity()` is always called with `await` but wrapped so it never throws. Primary operations must not fail because of logging.

10. **Batch DB operations.** Use `Promise.all` for parallel queries. Use `.insert(array)` for bulk inserts. Never loop with sequential `await` calls.

11. **Update this file** when making major architectural changes, adding new tables, changing business rules, or adding new routes.
